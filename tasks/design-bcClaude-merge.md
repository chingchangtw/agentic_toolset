## Problem Statement

The project root templates (`src/project_root_structure/`) lack three behavioral
patterns from Boris Cherny's proven CLAUDE.md approach:

1. No "Would a staff engineer approve this?" gate in the Definition of Done.
2. No explicit task management flow (plan → track → explain → capture).
3. No autonomous bug-fixing posture in the universal agent rules.

These gaps mean agents working from the templates have weaker self-correction loops
and no task discipline structure, reducing artifact quality over time.

## Target User & Narrowest Wedge

AI agent developers who bootstrap new projects using these templates.
Wedge: the three missing pieces that most concretely change AI behavior
in real sessions — not a wholesale rewrite of the files.

## Constraints

- Templates must stay under ~150 lines (existing CLAUDE.md template guideline).
- Governance file is already ~110 lines — can absorb ~15 more lines before hitting limit.
- BC's universal/behavioral content goes into governance + `.claude/CLAUDE.md`, NOT
  into the structural project-brief `CLAUDE.md`.
- No cross-file duplication: if a concept is in `.claude/CLAUDE.md` (universal), it
  must not repeat in governance.

## Premises

1. BC's behavioral patterns belong in `goverance_CLAUDE.md` + `.claude/CLAUDE.md`,
   not in the project-brief `CLAUDE.md` (which stays structural).
2. Three clearest gaps: (a) task management flow missing from governance, (b) autonomous
   fixing posture missing from universal, (c) self-improvement loop needs prominence.
3. The project-brief template doesn't need BC content — `tasks/` + lessons file already
   in the file map.
4. Doing nothing leaves agents without task-tracking discipline and self-correction loop.

## Approaches Considered

### Approach A: Minimal Surgical Patch (SELECTED)
Add exactly 3 targeted additions. No restructuring, no line deletions.

### Approach B: Full Systematic Merge
Map every BC concept to its proper home, strengthen 5 governance areas.
Larger diff, risk of scope creep past 150-line limit.

### Approach C: Behavioral Extraction
Encode BC's loop as a skill + hook instead of prose. Requires skill authoring
work; templates remain weaker until skills are built.

## Recommended Approach

**Approach A: Minimal Surgical Patch.**
Three edits, each in the right file. Preserves existing structure. Reviewable
as a single small PR.

## The Three Changes

### Change 1 — goverance_CLAUDE.md: Staff-Engineer Check in Definition of Done

**File:** `src/project_root_structure/.claude/goverance_CLAUDE.md`
**Location:** After the last bullet in `## Definition of Done`
**Add:**
```
- Ask: "Would a staff engineer approve this?" If no → revise before marking done.
```

**Why:** BC's most actionable self-check. Absent from current DoD. Forces a quality
bar that's concrete and repeatable, not vague.

---

### Change 2 — goverance_CLAUDE.md: Task Management Flow in Implementation Workflow

**File:** `src/project_root_structure/.claude/goverance_CLAUDE.md`
**Location:** After `## Implementation Workflow` section (after the 7-step list)
**Add block:**
```markdown
## Task Flow (per task)
1. Write plan to `tasks/todo.md` with checkable items.
2. Verify plan before starting implementation.
3. Mark items complete as you go — not in a batch at the end.
4. After corrections: update `.ai/LESSONS_LEARNED.md` with the pattern. Ruthlessly
   iterate until mistake rate drops.
```

**Why:** Current governance has a 7-step implementation workflow but no task-level
management discipline. BC's pattern fills exactly this gap. Note: "Review lessons at
session start" is already at line 110 of governance (`Self-Reflection Cadence`); do NOT
repeat it here.

**Line count check (pre-validated):** Governance is currently 110 lines. Change 1 adds
1 line; Change 2 adds 6 lines. Post-merge: ~117 lines — under the 150-line limit.

---

### Change 3 — .claude/CLAUDE.md: Autonomous Problem-Solving Posture

**File:** `src/project_root_structure/.claude/CLAUDE.md`
**Location:** At end of `## Core Principles` section (or as a new short section after it)
**Add:**
```markdown
## Autonomous Problem-Solving
- When given a bug report: fix it. Don't ask for hand-holding.
- Point at logs, errors, failing tests — then resolve them.
- Go fix failing CI tests without being told how.
- Don't stop at the surface symptom. Find the root cause and fix it completely.
```

**Why:** Current universal rules cover planning and verification but not the fixing
posture. BC's version is unusually direct — this is behavioral shaping content that
changes how agents approach open-ended debugging sessions. The last bullet replaces
BC's "Knowing everything I know now, implement the elegant solution" with a concrete
problem-solving directive that can't be misread as "rewrite for style."
