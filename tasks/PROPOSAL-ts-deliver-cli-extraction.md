## Problem Statement

`ts-deliver-router`'s phase logic (state transitions, gate enforcement, registry checks)
today lives inside a Claude-Code `Skill` — frontmatter-driven markdown that only Claude
Code's `Skill` tool can load and interpret. `.agents/*.json*` state files are plain
JSON/JSONL and already CLI-agnostic. The open backlog item
(`tasks/ideasToDoList.md` — "PRD — Multi-CLI Portability: Codex CLI + Copilot CLI
Targets") asks whether this project's dual-track workflow can extend beyond Claude
Code. The question forcing this session: does the skills+JSON foundation support that,
or does it need to become a "dynamic workflow" to get there?

## Demand Evidence

No incident has yet forced this — user confirmed the trigger is an "architectural itch,"
not a broken session. The itch has a real anchor, though: user is already hand-running
`jq`/`grep` against `.agents/state.json` and `phase-routing.ts` mid-session to check
routing state, which is exactly the workaround that shows up when logic isn't exposed
as a callable interface. That's this session's confirmed pattern — recorded as
`docs: require jq for .json/.jsonl content` in this repo's git history, same day as
this design doc.

## Status Quo

Workaround confirmed: manual `jq`/`grep` reads of `.agents/ts-deliver-router/state.json`
and `history.jsonl`, and reading `phase-routing.ts` by eye, when the running skill's
account of "where am I" needs double-checking. Cost: user's own time, and risk of
acting on stale state mid-session.

## Target User & Narrowest Wedge

Named human: the user themself, mid-session, is the one who pays when skills+JSON
state drifts or needs manual inspection — not a hypothetical downstream contributor
or end-user installer (both were offered, both declined in favor of this).

Narrowest wedge: a `ts-deliver` CLI subcommand (`ts-deliver status`, `ts-deliver
next`) that reads `.agents/ts-deliver-router/state.json` and returns the same answer
the skill currently computes from prose — callable by the user directly (`node
scripts/ts-deliver.mjs status`), no AI CLI required to invoke it.

## Constraints

- Must not break the existing `spine-consistency` test (`phase-routing.ts` and
  `work-unit-profiles.md` stay in lockstep) — locked in P3.
- `src/` is the shipped surface (packaged into `dist/release.zip`); any extracted
  engine that's meant to ship to end-users lives there, not in `.claude/skills/` or
  `.agents/`.
- No cross-skill imports rule (`CLAUDE.md` Hard Rule 2) — an extracted engine used by
  multiple skills needs a shared module under `src/`, not skill-to-skill imports.

## Premises

1. `.agents/*.json*` state is already portable (plain JSON/JSONL, jq-readable from any
   CLI) — **agreed**. Not the bottleneck.
2. The `Skill` tool + frontmatter-markdown loading is Claude-Code/gstack-specific.
   Codex CLI and Copilot CLI have no equivalent loader — this is the actual portability
   gap — **agreed**.
3. `phase-routing.ts`'s lockstep-tested spine is a feature worth preserving, not debt
   to dissolve into a fully dynamic/inferred workflow — **agreed**.

## Approaches Considered

### Approach A: Extract a CLI engine (recommended)
Pull `phase-routing.ts` + state-read/write/gate logic out of the skill into a
standalone `ts-deliver` CLI (Node/TS, this repo is already ESM). Each AI CLI platform
gets a thin wrapper: Claude Code's `SKILL.md` calls the binary instead of embedding
the logic in prose; Codex/Copilot get an instruction file or slash-command that shells
out to the same binary.
- Effort: M
- Risk: Low — `phase-routing.ts` already has no Claude-specific API surface; it's pure
  TS reading/writing JSON.
- Pros: state/gate/registry logic lives once, tested once — the 43 tests that
  directly exercise this logic (`phase-routing.test.ts`: 9, `state-schema.test.ts`:
  28, `spine-consistency.test.ts`: 6) move with it unmodified, since they test pure
  functions against JSON fixtures, not the skill-markdown layer; each platform
  wrapper is small and disposable.
- Cons: still requires writing N platform wrappers (one per CLI) — the wrapper problem
  doesn't disappear, it shrinks.
- Reuses: `phase-routing.ts`, `state-schema.test.ts`, `spine-consistency.test.ts`,
  `.agents/` JSON contract — all unchanged in shape.

### Approach B: MCP server
Wrap the router as an MCP server exposing tools (`get_state`, `advance_phase`,
`check_gate`). Any MCP-capable CLI calls it uniformly — no per-platform wrapper files.
- Effort: L — new protocol surface, server lifecycle, auth to build and maintain.
- Risk: Med — depends on Codex/Copilot's MCP client maturity, which is uneven today
  and outside this repo's control.
- Pros: zero per-platform wrapper code once built; matches where the ecosystem is
  heading.
- Cons: biggest lift, and the win (no wrappers) is contingent on external tooling this
  project doesn't control.

### Approach C: Keep skills, document platform gaps
No refactor. Ship a doc mapping what each platform can/can't consume today: `.agents/`
JSON — yes everywhere; skill-markdown routing — Claude Code only.
- Effort: S
- Risk: Low
- Pros: zero engineering risk, ships today.
- Cons: doesn't touch the itch; Multi-CLI Portability PRD stays blocked.

## Recommended Approach

**Approach A — extract a CLI engine.** It converts the itch into a testable artifact
(a binary carrying forward the 43 tests — of the project's 394 total — that directly
exercise this logic today) without betting on external MCP client maturity
(Approach B) or leaving the portability PRD unaddressed
(Approach C). The wedge (`ts-deliver status`/`next` as a callable CLI command) is
buildable and independently useful even before any second AI-CLI wrapper exists.

## Skill/CLI Boundary

Read-only operations move to the CLI binary: `status`, `next`, `check-registry`,
schema validation. Gate sign-off (`G1`/`G2` human approval) stays skill-side —
`never_automate.md` already requires a human-in-the-loop conversation for gates at
every autonomy level, and a CLI binary has no conversational surface to hold that
exchange. The binary can expose `check-gate` (read-only: is this gate satisfied?)
but not `sign-gate` (that stays a skill-mediated, AskUserQuestion-gated action).
This mirrors the existing DIAL/CHECKS-REGISTRY/DRY-RUN split in
`ts-deliver-router/SKILL.md` — nothing here overrides `never_automate.md`.

## Distribution Plan (resolved)

Ships under `scripts/` (dev-only), not `src/`, for this iteration. The problem this
doc addresses is *this repo's own* multi-CLI workflow, not a capability end-users
install — `tasks/ideasToDoList.md`'s Multi-CLI Portability PRD is about this
project's own dogfooding across Codex/Copilot, not a shipped feature. Moving it to
`src/` (and therefore `dist/release.zip`) is a separate, later decision once a second
platform wrapper actually exists and end-user demand for it is real — premature
today per this session's own P1/architectural-itch finding.

## Rollback / Failure Mode

Extraction is additive until cutover: the CLI binary is built and tested standalone
first; `ts-deliver-router/SKILL.md` keeps its current skill-markdown logic until the
binary's output is verified to match the skill's current behavior on a real
`.agents/` tree (same round-trip verification pattern used for
`pl-dogfood-activation` — before/after diff against a real state.json). If the binary
diverges, the skill-markdown path is simply left in place; no state schema changes
mean there's nothing to roll back in `.agents/*.json*` itself.

- Zero change to `.agents/*.json*` schema — the extraction touches the logic layer
