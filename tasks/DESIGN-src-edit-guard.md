# Design: src/ Edit Guard — block Claude Code edits to src/ without a matching Spectra change

Generated by /office-hours on 2026-07-09
Branch: master
Repo: chingchangtw/agentic_toolset
Status: DRAFT
Mode: Intrapreneurship (infra guardrail)

## Problem Statement

`src/` is the deliverable that gets packaged into `release.zip`. Nothing mechanically
stops a Claude Code session from editing `src/**` without an active, declared Spectra
change — which is exactly the propose→apply discipline the repo's CLAUDE.md demands
("Every change must solve a real, observed problem"; anti-slop rules). Today the
propose→apply flow is honored by convention only. The goal: make an unbacked `src/`
edit impossible to do silently inside a Claude Code session.

## Decisions locked (from office-hours)

- **Enforcement layer:** `PreToolUse` hook only. In-session, Claude Code only.
  Explicitly NOT covering humans-in-an-IDE or other CLIs (Codex/Copilot) in v1 —
  those are a separate, editor-agnostic problem (see Non-Goals / Open Questions).
- **Unlock rule:** the edited `src/` path must appear in the active change's declared
  file set ("Impact files"). A bare in-progress marker is not enough — the specific
  path must be accounted for.
- **On no-match:** do NOT hard-deny. Surface for **approval** (`permissionDecision:
  "ask"`) so the human can allow a one-off, rather than the agent being wedged.

## Premises (agree/disagree)

1. The guard governs the *authoring session only*. It is a speed-bump against silent
   scope drift, not a security boundary — anyone determined can bypass it (edit outside
   Claude Code, or approve the `ask`). That is acceptable for v1. — **agree/disagree?**
2. "A change in place" = a change with an in-progress marker at
   `.spectra/changes/<name>.started`. Merely having a non-archived `openspec/changes/<name>/`
   dir does NOT unlock `src/`. — **agree/disagree?**
3. The declared file set comes from the change's `proposal.md` `## Impact` section
   (Affected code → New/Modified/Removed paths), NOT from `.spectra/touched/*.json`
   (which is post-hoc, git-diff-derived, and full of unrelated paths). — **agree/disagree?**
4. Only `src/**` writes are guarded. Edits to `openspec/`, `tasks/`, `docs/`, `.claude/`,
   tests, and scripts pass through untouched — otherwise you could never author the
   proposal that unlocks the src edit. — **agree/disagree?**

## Grounding (what exists, do not rebuild)

- `.claude/settings.json` already runs `PostToolUse` hooks on `Edit|Write|Bash`. Adding a
  `PreToolUse` block is the same wiring, one phase earlier.
- Claude Code `PreToolUse` contract: hook gets JSON on stdin (`tool_name`,
  `tool_input.file_path`, …); it returns JSON on stdout with
  `hookSpecificOutput.permissionDecision` = `"allow" | "deny" | "ask"` +
  `permissionDecisionReason`. `"ask"` is exactly the "request approval" behavior wanted.
- In-progress state: `.spectra/changes/<name>.started` marker files. (`spectra in-progress`
  has only `add` — no `list` — so the hook scans the marker dir directly.)
- Declared files: `openspec/changes/<name>/proposal.md` `## Impact` section. Example from
  the change shipped earlier this session:
  `## Impact` → `- Modified: scripts/run-scenario.sh`.

## Approaches Considered

### Approach A: Impact-section allowlist (recommended)
  Summary: Hook resolves active in-progress change(s), parses each `proposal.md`
    `## Impact` block for repo-relative paths, unions them into an allowlist. If the
    Write/Edit target is under `src/**` and not in the allowlist → `ask`; if in the
    allowlist → `allow`; if not under `src/**` → `allow` (pass-through).
  Effort: S
  Risk: Low-Med
  Pros:
    - Uses the one file authored up front with intent (Impact = the declared blast radius).
    - Catches scope creep: editing a src file the proposal never mentioned triggers `ask`.
    - No dependency on spectra internals beyond the marker-file convention.
  Cons:
    - Impact paths must be accurate and repo-relative; a sloppy proposal under-declares
      and the author gets `ask`-nagged for legitimate edits (mitigation: `ask`, not `deny`).
    - Needs a small, robust path parser (Impact bullets, backtick-wrapped or bare).
  Reuses: `.spectra/changes/*.started`, proposal Impact convention, existing hook wiring.

### Approach B: Impact ∪ tasks.md backtick-paths allowlist
  Summary: Same as A, but also harvest any backtick-wrapped path in the change's
    `tasks.md` and add to the allowlist. Wider net for changes whose Impact is thin but
    whose tasks name concrete files.
  Effort: S-M
  Risk: Med
  Pros:
    - Fewer false `ask`s when authors put paths in tasks, not Impact.
  Cons:
    - tasks.md describes behavior, not just paths; backtick tokens are often commands,
      symbols, or globs, not files — noisier allowlist, weaker scope-creep signal.
  Reuses: same as A plus tasks.md.

### Approach C: Marker-only unlock (any in-progress change opens all of src/)
  Summary: If ANY `.spectra/changes/*.started` exists, allow all `src/**` writes;
    else `ask`. No per-path matching.
  Effort: XS
  Risk: Low (but weak)
  Pros:
    - Trivial hook; nearly zero false positives.
  Cons:
    - Fails the locked decision (path must match the change). A stale in-progress marker
      leaves src/ wide open. Rejected as primary — kept only as the degrade path if
      Impact parsing proves brittle.
  Reuses: marker files only.

## Recommended Approach

**Approach A**, with a hard-coded fallback to C's behavior only when the active change
has an empty/unparseable Impact (so a malformed proposal degrades to "marker unlocks
src" rather than blocking all work). This satisfies the locked decisions: PreToolUse,
path-must-match, `ask` on no-match.

### Hook logic (pseudocode — implementation is a follow-up change, not this doc)

```
on PreToolUse(Edit|Write|MultiEdit):
  path = tool_input.file_path (repo-relative)
  if path not under src/**:            decision = allow;  return
  active = [name for name in .spectra/changes/*.started]
  if active is empty:                  decision = ask
                                       reason = "No in-progress Spectra change. Run
                                                 /spectra-propose or /spectra-apply first."
                                       return
  allow = union(parse_impact(openspec/changes/<name>/proposal.md) for name in active)
  if allow is empty:                   decision = allow   # malformed-proposal degrade (C)
  elif path in allow:                  decision = allow
  else:                                decision = ask
                                       reason = "<path> is not in the Impact of active
                                                 change(s) <active>. Add it to the proposal
                                                 or confirm this one-off edit."
```

`parse_impact` extracts repo-relative paths from the `## Impact` section: lines under
`Affected code:` / `New:` / `Modified:` / `Removed:`, stripping backticks and list
markers. Non-path prose is ignored.

## Success Criteria

- Editing a `src/**` file NOT named in any active change's Impact → Claude Code shows an
  approval prompt with a reason naming the path and the active change(s).
- Editing a `src/**` file that IS in the active change's Impact → silent allow.
- Editing outside `src/**` (openspec/, tasks/, tests, scripts, docs) → silent allow.
- No active in-progress change + any `src/**` edit → approval prompt pointing at
  `/spectra-propose` / `/spectra-apply`.
- Guard adds < ~50ms to each Edit/Write (marker scan + one proposal parse; cheap).

## Distribution Plan

The hook is a Claude Code artifact. Two homes, and the choice is an open question:
- **This repo's dev env** (`.claude/hooks/` + a `PreToolUse` block in
  `.claude/settings.json`) — dogfood only, governs sessions editing THIS repo.
- **Shipped to end users** (`src/hook/` + release-manifest entry, installed to a target
  project's `.claude/`) — makes the guard part of the toolset product, governing any
  project that installs it and uses Spectra. Bigger blast radius; needs the guard to be
  correct against arbitrary project layouts (src glob may differ). See OQ-3.

## Open Questions

1. **Multiple in-progress changes** — union all their Impact allowlists (current design),
   or require exactly one active change (simpler mental model, but blocks parallel work)?
2. **`src/` definition per project** — hard-code `src/**`, or read a configurable guarded
   glob (some installed projects put deliverables elsewhere)? Ship-to-users forces this.
3. **Ship in-repo only vs. as a product hook** (Distribution Plan). Recommend in-repo
   dogfood first; promote to `src/hook/` once it survives a few real changes here.
4. **Editor-agnostic layer later?** v1 is AI-session only (locked). If humans/other CLIs
   need the same guard, a git pre-commit hook parsing the same Impact allowlist is the
   natural R2 — same parser, different trigger. Out of scope now.

## The Assignment

Before writing any hook code: take the *next* real `src/` change you make and, by hand,
check whether its `proposal.md` `## Impact` section lists every src path you actually
touch. If it doesn't (Impact under-declares in practice), Approach A will `ask`-nag you
constantly and you want Approach B or a stricter proposal-authoring habit instead. One
real data point decides A vs B before you build either.

## What I noticed

- You tightened the ask twice: first "restrict changes to src", then "by exactly Spectra
  Change and Tasks", then "if any, ask for approval." You're designing a speed-bump, not
  a wall — the `ask` escape hatch is doing deliberate work, not a compromise.
- You picked the strictest unlock rule (path must match) but the softest failure mode
  (ask, not deny). That combination is the right instinct for a guardrail you'll be
  living inside daily: high signal, low lockout.
